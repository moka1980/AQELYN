1. Executive Summary

The AQELYN Identity Security Posture Management Intelligence Engine provides the identity posture intelligence layer for the AQELYN Cyber Security Operating Environment. It continuously discovers identities, normalizes source data, evaluates posture, calculates risk, detects drift, maps findings to compliance obligations, and recommends remediation. The engine does not act as an identity provider and does not directly mutate external identity systems. Instead, it produces trusted posture intelligence, evidence, events and remediation instructions for downstream AQELYN services.

The purpose of EA-0033 is to provide enough engineering detail for implementation teams to build the engine consistently after the broader AQELYN architecture is complete. The package therefore includes normative requirements, architecture components, data models, event contracts, API contracts, security controls, lifecycle rules, performance design, validation strategy, traceability, examples and diagrams.

The archive is designed to be implementation-ready. A team can derive backlog items, source modules, unit tests, integration tests and acceptance tests directly from this package.

2. Scope

2.1 In Scope

  • Identity discovery from enterprise, cloud, workload and platform identity sources.
  • Normalization into AQELYN Universal Object Model identity records.
  • Identity classification, ownership and lifecycle state tracking.
  • Authentication, authorization, monitoring, governance and lifecycle posture checks.
  • Identity posture scoring and risk categorization.
  • Drift detection against approved identity baselines.
  • Compliance and policy mapping.
  • Evidence association for every material posture finding.
  • Event publication to AQELYN Event Bus.
  • Knowledge Graph relationship updates.
  • Remediation recommendation generation.
  • Historical analytics for trend, delta and effectiveness reporting.

    • 2.2 Out of Scope

  • Direct modification of identity provider configuration.
  • Replacement of existing IAM, IGA, PAM, CIEM or SIEM products.
  • User authentication flows for AQELYN itself.
  • Secret vault implementation.
  • Manual governance approval workflows, except as integration targets.

    8. Scoring Model

    The Identity Security Posture Score is a deterministic 0-100 score where 100 means no detected identity posture weakness under the active rule set. Scores are calculated from weighted dimensions.

    | Dimension | Weight | Examples |

    | --- | --- | --- |

    | Authentication Quality | 25% | MFA, phishing-resistant authentication, credential age, legacy auth |

    | Authorization Quality | 25% | Admin roles, excessive permissions, toxic combinations |

    | Lifecycle Hygiene | 15% | Dormant, orphaned, expired, ownerless accounts |

    | Governance and Compliance | 15% | Policy violations, exceptions, approvals |

    | Monitoring Coverage | 10% | Audit logs, identity telemetry, detection coverage |

    | Behavioral and Trust Signals | 10% | Trust score, anomaly hints, mission context |

    | Score Range | Rating | Required Response |

    | 75-89 | Good | Planned improvement |

    | 60-74 | Moderate | Remediation backlog |

    | 40-59 | Elevated | Priority remediation |

    11. Security Architecture

    The ISPM engine processes sensitive identity metadata. Security controls therefore apply to connectors, storage, APIs, events, logs and generated artifacts.

    11.1 Core Controls

  • Least-privilege provider credentials scoped to read-only posture assessment wherever possible.
  • Secret material stored only in an approved secrets facility, never in logs or events.
  • Encryption in transit for all service calls.
  • Encryption at rest for inventory, assessment history and evidence references.
  • Field-level redaction for sensitive attributes in logs.
  • Role-based authorization for API and administrative operations.
  • Separation of connector execution roles from scoring and publication roles.
  • Immutable audit trail for baselines, rules and assessment outputs.

    • 11.2 Abuse Resistance

    The engine must not expose credential material, must not provide instructions for privilege escalation, and must not execute remediation against external identity providers unless a separate approved workflow engine action authorizes the change.

    12. Lifecycle Architecture

    12.1 Discovery Lifecycle

    1. Connector job scheduled or triggered.

    2. Provider queried using cursor or full sync.

    3. Payload validated and normalized.

    4. Inventory updated.

    5. Change events emitted.

    12.2 Assessment Lifecycle

    1. Assessment scope selected.

    2. Applicable policies, baselines, rule sets, trust inputs and mission context loaded.

    3. Posture checks executed.

    4. Score calculated.

    5. Findings and evidence references persisted.

    6. Events published.

    7. Recommendations generated.

    12.3 Drift Lifecycle

    1. Current identity state retrieved.

    2. Approved baseline resolved.

    3. Comparator identifies deviation.

    4. Drift severity assigned.

    5. Drift finding emitted with evidence.

    6. Recommendation linked.
