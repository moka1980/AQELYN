Vision
Every observation inside AQELYN becomes evidence.
Evidence is:
•  Verifiable
•  Traceable
•  Immutable (where appropriate)
•  Searchable
•  Versioned
•  Explainable
•  Linked to objects
•  Linked to missions
•  Linked to findings
Evidence is never separated from the object that produced it.

Chapter 7
Evidence Confidence
Each evidence source is evaluated independently.
Example:
•  Direct API response → High confidence.
•  Verified system API → High confidence.
•  Passive observation → Medium confidence.
•  Incomplete observation → Lower confidence.
The confidence reflects the quality of the evidence, not the severity of the issue.

PS-ADR-0019


--- Page 461 ---

Evidence Is the Source of Truth
Status: Accepted
Decision
Every  security  conclusion  inside  Project  AQELYN  shall  be  traceable  to  one  or  more
evidence objects.
Rationale
Evidence-first design improves transparency, auditability, and user trust.
NEW CORE ENGINE
Evidence Correlation Engine 2.0
Responsibilities:
•  Merge duplicate observations.
•  Detect conflicting evidence.
•  Rank evidence quality.
•  Link evidence to related objects.
•  Maintain evidence history.
This improves reasoning while preserving original observations.
NEW CORE ENGINE
Evidence Replay Engine
A unique capability.
Users  can  replay  the  exact  evidence  collected  during  a  mission  to  understand  how
conclusions were reached and to compare historical states.
NEW CORE ENGINE
Evidence Integrity Monitor
Continuously verifies:
•  Hash consistency.
•  Metadata integrity.
•  Missing evidence.
•  Broken relationships.
•  Storage health.
Alerts are generated if integrity checks fail.
NEW IDEA


--- Page 462 ---

Cyber Security Flight Recorder
Inspired by aviation.
AQELYN continuously records operational events during a mission.
If a major incident occurs, investigators can replay:
•  Discoveries.
•  Scan execution.
•  Evidence collection.
•  Correlation.
•  Decisions.
•  User actions.
•  Remediation.
•  Verification.
The Flight Recorder becomes invaluable for incident response and post-incident reviews.
NEW IDEA
AQELYN Evidence Notebook
Every mission automatically creates a structured notebook where analysts can:
•  Add observations.
•  Attach evidence.
•  Record hypotheses.
•  Document conclusions.
•  Link related objects.
•  Track investigation progress.
This combines technical evidence with human expertise, making investigations easier to
understand and audit.
NEW IDEA
Evidence Trust Index (ETI)
Every mission receives an Evidence Trust Index based on factors such as:
•  Coverage of the assessed environment.
•  Diversity of evidence sources.
•  Freshness of collected data.
•  Integrity verification.
•  Correlation quality.
•  Verification success.
The  ETI  indicates  how  well-supported  the  mission's  conclusions  are,  helping  users
understand the reliability of the assessment.

The  next  engineering  milestone  is  the  Web  Security  Engine,  one  of  the  flagship
components of AQELYN.
Unlike  many  scanners,  it  will  not  simply  enumerate  issues.  It  will  be  organized  around
reusable analysis engines such as:
•  HTTP Analysis Engine.
•  TLS Analysis Engine.
•  Header Analysis Engine.
•  Content Security Policy Analyzer.
•  Authentication Assessment Engine.
•  Technology Fingerprinting Engine.
•  API Assessment Engine.
•  Passive Discovery Engine.
•  Authorized Active Assessment Engine.
•  Screenshot and Visual Change Engine.
•  JavaScript Analysis Engine.
•  Evidence Collection Layer.
•  Trust Evaluation Layer.
Each  engine  will  produce  structured  evidence,  update  the  Knowledge  Graph,  and
contribute to the overall Trust Profile.
A final proposal for our engineering philosophy
I believe we should adopt one overarching design rule that governs every implementation
decision:
AQELYN never hides complexity—it organizes it.
Internally, the platform can be extremely sophisticated, but externally it should present
users  with  clear  missions,  evidence-backed  findings,  understandable  recommendations,
and guided workflows.
If  we  remain  faithful  to  that  principle,  we  can  build  a  Cyber  Security  Operating
Environment  that  is  powerful  enough  for  security  professionals  while  remaining
approachable  for  home  users,  consultants,  enterprises,  financial  institutions,  and
government organizations.
